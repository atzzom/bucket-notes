HTML — Building a table's structure

As expected tables are introduced when is necessary display structured data, however, the situation doesn't always been like that, in the first days of HTML, tables were also used to position content and as an aid to the styling of pages; the situation came with its tradeoffs, because this was not the table's purpose. Now, tables are devoted to its intended design, leaving to the CSS properties the styling and positioning tasks.

How a table should be built in HTML largely depends on the contained data, legibility criterion, and the desired layout. To semantically mark up the columns and rows a table must have, the minimum HTML to be used are: <table>, <tr> (table row), and <td> (table data). Together, with additional elements such as <th> (table header), <tfooter> among others will result in a solid layout.

Table

The <table> element is used to semantically initialize or mark up the beginning of a tabular arrangement, acting as the parent element of all table elements.

Table row

This element determines the content in a table row. A table can have numerous rows—<tr> elements— to fit the information to display.

Table data

Only once the <table> and the <tr> are declared, data cells can be added using the <td> element. Listing multiple <td> elements one after the other within a table row will create a column layout.

Table header

This element is represented with the <th> tag, is works as a semantic highlighter for a cell, giving it higher hierarchy in comparison to a regular <td>. It is possible to specify whether a <th> applies to a row or to a column with the scope attribute, which can take the values: col, row, colgroup, and rowgroup; for example, the <td scope="col">Title>/td> indicates that this header applies for all the cells directly below it—or in the same column—whereas in a students' grade list, the element <th scope="row">Name</th> indicates that every cell within the row relates directly to that table header. Additionally, depending on the browser, table headers may receive some default styling, usually bold and centered.

Using the <th> element, along with the scope attribute, tremendously helps screen readers and assistive technologies make sense of a table. Therefore, they are also benefical to those browsing a web page using those technologies.

Table: head, body, & foot

Mocking the structure of a web page, tables also count with a head: <thead>, body: <tbody>, and footer: <tfoot> that help to structurally organize tables.

The <thead> element must come immediately after any <caption> element, its purpose is to (semantically) mark up the heading rows of the table.

Prior to the introduction of the HTML5, the element that must come immediately after the <thead> was the <tbody>, but now the new standard has provided leeway here, allowing the use of the <tfoot> element after the <thead> maybe because of tables with only header and footer?. The only consideration is guarantee, neither of them (the <tbody> nor the <tfoot>) is parent element of the other. The <tbody> element should contain the primary data within a table, while the <tfoot> element semantically contains data that outlines the contents of a table.

CSS — Table styling

So far, the elements that provide structure to a table have been outlined, but although the information is organized, specially when dealing with large amounts of data (columns and/or rows) becomes increasingly important the legibility of it. This purpose (overall legibility, and overall design) is achieved modifying CSS parameters, explained below.

Table borders — border collapse

Each of the different table elements has borders that start to stack up , with the border of one element sitting next to that of another element. For example, if we put a 2–pixel border around each table cell, there would be a 4–pixel border around every cell in the table.

To define the border model of a table the border–collapse property is used, with the available collapse, separate, and inherit. By default, the border–collapse property value is separate, meaning that all of the different borders will stack up next to one another, as described above. The collapse property, on the other hand, condenses the borders into one, choosing the table cell as the primary border.

Table borders — border spacing

As the border–collapse property with the separate value allows borders to be stacked up against one another, the border–spacing property can determine how much space, if any, appears between the border.

For example, a table with a 1–pixel border around the entire table and a 1–pixel border around each cell will have a 2–pixel border all around every cell because the borders stack up next to one another. Adding in a border–spacing value of 4 pixels separates the borders by 4 pixels.

The border–spacing property works only when the border–collapse property is set to its default value, i.e. separate. In the case that the border–collapse property had not been declared, the browser will assume this default value should any border be applied to the table, hence, in such a case, the border–collapse property can be used without worry.

Additionally, the border–spacing property accepts two length values: the first one for horizontal spacing and the second one for vertical spacing. Therefore, the declaration border–spacing: 5px 10px; will place 5 pixels of horizontal spacing between borders and 10 pixels of vertical spacing between borders.

Adding borders to rows

As is not possible to add borders to the table's structural elements, a border–bottom cannot be applied to a <tr> element. To apply a row border it is possible to follow this process:

  1. Set the border–collapse property to collapse so borders are not staked up between them.
  2. Add a border–bottom property cell by cell, it doesn’t matter if it is a <td> or a <th>.
  3. Optionally, it is possible to remove the bottom border from the cells within the last row of the table by using the :last-child pseudo-class selector to select the last <tr> element within the table and target the <td> elements within that row. Additionally, if a table is using the structural elements, is necessary prequalify the last row of the table as being within the <tfoot> element.

Table Striping

In the effort to make tables more legible, one common design practice is to "stripe" table rows with alternating background colors. This makes the rows clearer and provides a visual clue for scanning information. In order to achieve this pattern, a class element can be used in the desired striped rows, or, alternatively, use the :nth–child pseudo-class selector with an even or ood argument to select every other <tr> element.

In the previous code, the <table> element has an explicit border-collapse property se to separate and the border-spacing is set to 0. The reason for this is that the <td> elements include border while the <th> elements do not. Without the border-collapse property set to separate the borders of the <td> elements would make the body and foot of the table wider than the head.

Since the border-collapse property is set to separate is necessary to be careful as to how borders are applied to <td> elements. Only the first <td> element will receive a border-left.

Aligning text

The text layout also plays an important role in improve the legibility of the information. Commonly, names, descriptions, and so forth are flush left, while numbers and other figures are flush right. And depending on its context and/or relevant some text could be centered.

As previously seen, the horizontal alignment can be achieved using the text-align CSS property. In regard to the vertical alignment, the vertical-align CSS property is used; is necessary to note that this property applies for inline elements only, accepting the top, middle, and bottom keyword values. To apply the desired alignment to cells, a class can be added and styled accordingly using CSS.
